## 0. The pitch in one paragraph

> **madcop** is a personal AI agent for AI PMs (Product Managers working on AI products).
> You talk to it from your terminal, it talks to any LLM you point it at, it remembers
> what you taught it last time, and it gets smarter the more you use it.
> It's a **general-purpose agent at the core** + **a supply chain skill pack on top** —
> because the author lived through 4 years of supply chain OMS/WMS/TMS pain
> and figured out that the only difference between a generic agent and a useful one
> is **4 layers of memory that grow with you**.

---

## 1. Why (the "because I lived it" story)

**AI PMs (Product Managers working on AI products) jump between 5-8 tools every day** —
Cursor, Claude Code, note apps, chat tools, spreadsheets, internal dashboards,
customer CRMs, ad-hoc SQL. Every switch costs attention and breaks flow.

**madcop is one agent that talks to all of them.** It runs in your terminal, points
at the data, and gives you a structured report in 60 seconds.

**Why I built it**: I worked in supply chain solutions for 4+ years, building
OMS / WMS / TMS integrations across 40+ external systems for large enterprise
customers. Every project had the same pain: data lives in 5 systems, every
diagnostic takes 30 minutes of manual stitching, and every new question is a
new investigation. I built madcop so an AI agent can answer those questions
in 60 seconds.

**Why supply chain domain knowledge in a "general" agent**: Because nobody else
will. Claude / Cursor / Hermes are great at generic tasks but they don't know
what a CUSUM is, what an OTWB flow looks like, or why a frozen-zone breach
matters more than an ambient-zone one. madcop is **general-purpose at the
core, supply-chain-smart at the skill layer**. Use it for any task; install
the supply chain skill pack when the task touches logistics.

## 3. User stories (the "what success looks like")

### US-1: Diagnose a supply chain anomaly
> "I want the agent to tell me why order cancellations spiked yesterday."

- **Before madcop**: Manually pull 4 SQL tables → Excel pivot → write a paragraph (30 min)
- **With madcop v0.6.0**:
  ```bash
  madcop run agent "Why did order cancellations spike yesterday?" \
    --adapter oms --days 1
  ```
  → **1 minute** returns: 5 findings, 2 recommended interventions, cost estimate

### US-2: Turn meeting notes into OKRs
> "I want the agent to extract OKRs from this meeting transcript."

- **Before madcop**: Manually read + tag + rewrite + paste into note app (15 min)
- **With madcop v0.6.0**:
  ```bash
  madcop run agent "Extract OKRs from this meeting" --input notes.md
  ```
  → **30 seconds** returns: 3-5 OKRs with owner / metric / deadline

### US-3: Draft a weekly status update
> "I want the agent to draft my weekly status from this week's events."

- **Before madcop**: 1 hour per week of manual writing
- **With madcop v0.6.0**:
  ```bash
  madcop run agent "Write this week's status" --template weekly.md \
    --input events.jsonl
  ```
  → **5 minutes** returns: 80% draft, human reviews and edits

### US-4: Cost-aware LLM routing
> "I want the agent to pick the right model automatically, but let me override."

- **Auto mode** (default): agent picks cheap model for trivial tasks, expensive
  model for reasoning-heavy tasks. Tracks cost in real time.
- **Manual mode**: User configures which model for which task type in `~/.madcop/config.yaml`.

## 4. v0.6.0 scope (and explicit non-goals)

### ✅ IN scope

1. **Multi-provider LLM client** — pluggable base URL, supports OpenAI-compatible
   APIs (OpenAI, NVIDIA NIM, GLM, DeepSeek, Qwen, etc.)
2. **Auto model routing** — classify task difficulty, route to cheap/expensive model
3. **4-layer memory** — working / episodic / semantic / reflective
4. **Self-growth engine** — episodic → semantic distillation, feedback loop, meta-pattern mining
5. **Scratchpad persistence** — agent state written to disk between steps;
   resumable across crashes
6. **Context compaction** — summarize every N steps to keep prompt under budget
7. **Multi-step agent loop** — plan → execute → verify → replan (with crash recovery)
8. **Cost tracking** — real-time $ spent per run, persisted to scratchpad
9. **1 demo end-to-end** — a 5-8 step supply chain diagnostic that proves the loop
10. **Benchmark** — compare v0.5.0 (no routing + 1-layer memory) vs v0.6.0 (routing + 4-layer memory) on:
    - Token cost per run
    - Final answer quality (golden-set scoring)
    - Wall-clock time
    - "Memory growth" — # of semantic facts extracted, # of meta-patterns learned

### ❌ NOT in v0.6.0 (explicit non-goals)

1. **No web UI** — CLI only; a Web UI is a v0.7.0 discussion
2. **No new supply chain domain logic** — reuse v0.5.0's anomaly / replay / RCA /
   counterfactual; v0.6.0 is the *agent* wrapping them, not new domain math
3. **No multi-agent collaboration** — single agent loop, not openclaw-style
   Writer/Editor/Reader. That's v0.8.0.
4. **No fine-tuning or custom model training** — pure prompt engineering + routing
5. **No vendor lock-in** — every LLM call goes through the OpenAI-compat client
6. **No 3rd-party API for non-LLM services** (no Slack, no email, no calendar in v0.6.0)
7. **No team collaboration features** — single user, single machine
8. **No enterprise SSO / RBAC** — N/A for personal use

## 5.5  The 4-layer memory system (the "growing child" architecture)

This is the heart of v0.6.0. Most agents have one or zero memory layers;
madcop has four, each with a distinct role, modelled on human cognition.

```
┌────────────────────────────────────────────────────────────────┐
│  L1  Working Memory  (existing v0.5.0 ConversationBuffer)      │
│      "What I'm thinking about right now"                       │
│      Storage: in-memory deque / Scope: current session          │
│      Capacity: 50 messages OR 8K tokens                        │
│      Lifetime: cleared at session end                          │
├────────────────────────────────────────────────────────────────┤
│  L2  Episodic Memory    🆕 v0.6.0                              │
│      "What I've done" — every task: (goal, steps, outcome)     │
│      Storage: SQLite / Scope: cross-session / Indexed: time+goal│
│      Capacity: unbounded (rotated at 10K episodes)             │
│      Use: "Last time the user asked a similar question, I did…" │
├────────────────────────────────────────────────────────────────┤
│  L3  Semantic Memory    🆕 v0.6.0                              │
│      "What I know" — distilled facts/concepts/relations         │
│      Storage: SQLite + tag index / Scope: cross-session         │
│      Capacity: 1-10K facts                                    │
│      Use: "CUSUM threshold for ARL0=1000 in pharma = 4.78"    │
├────────────────────────────────────────────────────────────────┤
│  L4  Reflective Memory  🆕 v0.6.0                              │
│      "What I've learned about how to work"                     │
│      Storage: SQLite / Scope: cross-session                     │
│      Capacity: 100-500 entries                                │
│      Use: "User prefers ≤ 5 bullet summaries"                 │
│      ★ THIS is the "growth" engine — the difference between   │
│        a tool and a colleague.                                 │
└────────────────────────────────────────────────────────────────┘
```

### Three growth mechanisms (the "child growing up" loop)

**Mechanism 1 — Episodic → Semantic distillation**
```
After every task:
  episodic["Diagnose OMS cancel spike"] = {steps, findings, outcome}
  ↓ LLM distills
  semantic["OMS cancel spike common causes"] = [...3-5 candidates...]
  semantic["Diagnostic cost = ¥0.03 baseline"] = "Avg across 12 runs"
```
*The agent compresses experience into reusable knowledge.*

**Mechanism 2 — Reflective feedback loop**
```
After every task, user rates 1-5★ (optional but encouraged):
  ↓
  LLM reflects: "User gave 3★ — analysis too long, missed cost"
  ↓
  reflective["user preferences"] += "Summary ≤ 5 bullets, include cost"
  ↓
  Next task: prompt auto-injects these preferences
```
*The user "educates" the agent through feedback.*

**Mechanism 3 — Self-improvement meta-pattern mining**
```
After every N tasks (default 5), agent asks itself:
  "Where did I waste steps? What worked? What failed?"
  ↓
  Extract meta-patterns:
    - "If task keyword = '诊断', use rule-based CUSUM first, then LLM"
    - "If task keyword = '写文档', generate outline first, then fill"
  ↓
  Write back to episodic with marker `kind=meta-strategy`
  ↓
  Next similar task: planner consults meta-strategies first;
                     if hit, skip re-planning and execute directly
```
*The agent improves its own working methods.*

### Memory retrieval — FTS5 (full-text search)

- **v0.6.0 storage**: SQLite with FTS5 (built into Python's `sqlite3` stdlib)
- **v0.6.0 retrieval**: keyword + time filter (no semantic similarity yet)
- **v0.7.0 upgrade path**: swap FTS5 for `sqlite-vec` (vector similarity) without API change

**Why not ChromaDB / pgvector / Weaviate?**
- Extra dependency, extra docker, extra cost
- v0.6.0 is single-machine single-user — FTS5 covers 90% of needs
- "Premature optimization" — v0.6.0 proves the loop, v0.7.0 adds vectors

## 5.6  Model routing — the "auto-mode" decision table

When `routing.auto: true` (the default), the router scores each task 0.0-1.0
and picks one of three tiers. The user can override in `~/.madcop/config.yaml`.

### Tier table

| Tier | Purpose | Default provider | Cost (input per 1M) |
|------|---------|------------------|---------------------|
| **T1 Strong** | Plan, reflect, complex reasoning, meta-pattern mining | NVIDIA NIM `zhipu/glm-5` | ~$0.50 |
| **T2 Fast** | Execute, tool calls, format, parse, summarise | NVIDIA NIM `minimax/minimax-m3` | ~$0.10 |
| **T3 Local** | Rule checks, JSON parse, schema validate, regex | Local (no API call) | $0 |

### Difficulty score (sum to ~1.0)

| Signal | Weight | Notes |
|--------|--------|-------|
| Task requires planning ("分析"/"设计"/"规划") | +0.30 | Single biggest signal |
| Open-ended goal ("帮我想想") | +0.20 | Ambiguity → needs strong model |
| Multi-step ("然后"/"接着"/"之后") | +0.15 | Sequential reasoning |
| Touches supply chain domain (WMS/OMS/TMS/CUSUM/etc.) | +0.30 | Domain knowledge required |
| Touches finance / legal / medical | +0.25 | High-stakes domain |
| Touches code | +0.15 | Moderate reasoning |
| Context > 10K tokens | +0.20 | Long context reasoning |
| User keyword "重要"/"关键"/"must be accurate" | +0.10 | Explicit priority |
| Simple lookup / parse / format | −0.30 | Strong negative signal |

**Bucket rule** (final tier):
- score ≥ 0.70 → **T1 Strong**
- 0.40 ≤ score < 0.70 → **T2 Fast**
- score < 0.40 → **T3 Local** (skip LLM call)

### Per-agent-step routing (the actual call site)

| Agent step | Default tier | Reason |
|------------|--------------|--------|
| `planner.decompose(goal)` | **T1** | Sets the whole run up |
| `executor.run(subtask)` | **T2** | Tool-flavored, structured |
| `verifier.check(subtask, output)` | **T3 first** | 80% rule-checkable |
| `verifier.fallback_ambiguous` | **T2** | When rule fails |
| `reflector.extract_lesson` | **T1** | Meta-cognition |
| `aggregator.summarize` | **T1** | Output quality sensitive |
| `replanner.after_failure` | **T1** | Reasoning about failure |
| `json_parser / schema_validate` | **T3** | Pure parse, no LLM |

### User override format (`~/.madcop/config.yaml`)

```yaml
models:
  planner: nvidia/zhipu-glm-5
  executor: nvidia/minimax-m3
  verifier_fallback: nvidia/minimax-m3
  reflector: nvidia/zhipu-glm-5

routing:
  auto: true                     # if false, use manual_routing below
  cost_budget_per_run: 0.50      # CNY, hard cap
  context_budget_per_call: 30000 # tokens

# Used only when routing.auto = false
manual_routing:
  plan:        T1
  execute:     T2
  verify:      T3
  reflect:     T1
  aggregate:   T1
  replan:      T1
```

### Cost guardrails

- `cost_budget_per_run: 0.50` → agent refuses to start a step that would push
  total over ¥0.50, surfaces "over budget" with manual-override flag
- `context_budget_per_call: 30000` → context compactor kicks in earlier than default

## 9. Risk register (what could go wrong)

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| NVIDIA NIM rate-limits during demo | Medium | Demo fails | Cache response locally; allow `MADCOP_OFFLINE=1` |
| Auto router misclassifies task → wrong model | Medium | Bad answer quality | Default to safe model (NVIDIA minimax); let user override |
| Context compactor loses critical info | Low | Bad answers | Golden-set regression test that asserts critical info survives compaction |
| Scratchpad file grows unbounded | Medium | Disk full | Implement rotation + size cap in v0.6.0 |
| PyPI publish fails | Low | No public release | Local wheel is enough for P0/P1 users; PyPI is P2 |
| GitHub push 443 timeout | High (observed) | Delay | Use HTTP/1.1 + long timeout, retry 3-5 times |

## 12. What v0.6.0 is NOT

This section exists to prevent scope creep.

- **It's not a competitor to Cursor / Claude Code.** It's a CLI that runs in
  your terminal *alongside* them. Use Cursor for code; use madcop for ops/diagnostics.
- **It's not a multi-agent framework.** Single agent loop only. No Writer/Editor/Reader
  pipeline. (That comes in v0.8.0 if there's demand.)
- **It's not a SaaS product.** No hosted version, no accounts, no billing.
- **It's not a new supply chain domain library.** All domain math stays in
  v0.5.0. v0.6.0 only adds the *agent* wrapping them.
- **It's not a research project.** No benchmarks against academic SOTA.
  Engineering pragmatism over novelty.
